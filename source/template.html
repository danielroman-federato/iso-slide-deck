Ops Dashboard

INGEST
BUILT FOR
●Implementation engineers loading a fresh ISO ERC delivery
●Data-operations teams running bureau update pipelines
●Actuaries & PMs validating rates and factors are correct
THE PROBLEM

ISO delivers rate content state by state. Federato builds a shared library of ingested ERC content — rates, factors, territory definitions, rules, forms and bureau test cases — that every carrier onboarding draws from. Done manually it is error-prone and non-repeatable; done as a managed library it is the asset every downstream stage of integration depends on.

WHAT THE PROTOTYPE DOES
Mass ingest UI — point at a bureau delivery and see what is loadable before committing
Live per-state progress — each delivery's status is visible as it loads, not buried in logs
Per-delivery outcomes — what loaded, what failed, what was skipped, with the reasons surfaced
Bureau correctness check — the bureau's own test cases run against the loaded content; pass/fail at a glance
Active edition view — what is live in each state and when it became effective
WHAT THIS MEANS IN PRACTICE
INITIAL ONBOARDING
→
A multi-day database wrestle becomes a one-screen workflow — implementation engineers stop firefighting bureau deliveries
→
New customers onboard against a validated library — not a fresh from-scratch load every time
LONG-TERM MANAGEMENT
→
One library serves every customer — no per-tenant duplication, no drift between customers
→
Edition refreshes are routine — anyone sees what is live in each state and as of when, without writing a query

Customers operate multiple paper companies, each with its own product mix, state footprint and filings. Without a hub where customers manage their integration, those scopes get hardcoded, state activation lives in spreadsheets, and a new paper company means code changes.

Deviation Manager

DEVIATIONS
BUILT FOR
●Actuaries defining and calibrating carrier deviations
●Product managers reviewing and approving submissions
●Compliance officers tracking filed-vs-active deviations
●Underwriting managers seeing the live pricing and forms levers
THE PROBLEM

Carriers rarely use bureau content unchanged. The most consequential departure is forms — proprietary carrier forms replacing ISO base forms and endorsements, per state and subline. Rate departures layer on the same governance: LCMs, fees, schedule and tier factors, eligibility overrides. The hard part is not any single override — it is having one governed place to manage any deviation (form or rate), see what is actually live versus filed, and trace it through to the issued policy with a record of who approved what.

WHAT THE PROTOTYPE DOES
Proprietary form substitution (primary use case) — replace any ISO base form or endorsement with a carrier-proprietary form, scoped by state and subline; the bound policy reflects the override automatically
Rate deviations on the same rails — LCMs, fees, schedule mods, tier factors and algorithm overrides resolve to overlay steps in the rating algorithm and show up in the step trace
One governed framework for any departure — form substitutions and rate overlays are instances of one mechanism: same approval flow, same audit, same per-carrier scoping
Per-state, per-subline scoping — every deviation applies to chosen states and sublines; the matrix shows what carries which value
Approval workflow & bulk fill — drafts move through a configurable review cycle with stalled items flagged; apply one value across a state row or column
WHAT THIS MEANS IN PRACTICE
→
Actuaries stop maintaining shadow spreadsheets — "what we filed vs. what is live" is one source of truth
→
Customers run their proprietary forms confidently — the forms that differentiate them, without rebuilding the engine
→
Filing teams track every pending deviation against its regulatory deadline — in one place, not in calendars and inboxes
→
A new deviation goes from spec to live in days, not quarters — same governance, faster cycle
→
Examiners get clear answers — what changed, when, who approved, what filing it ties to

Algorithm View

ALGORITHM
BUILT FOR
●Actuaries reviewing the step-by-step rating algorithm
●Engineers adding a carrier-specific overlay step
●PMs confirming the bureau sequence matches the ISO manual
THE PROBLEM

The rating algorithm is a long ordered sequence of steps with strict bureau-defined ordering. Adding a carrier-specific overlay means inserting at exactly the right position without disrupting the bureau base. Without a visual view, actuaries can't verify bureau correctness, and the algorithm is opaque to anyone not reading code.

WHAT THE PROTOTYPE DOES
Step-by-step rendering — the canonical algorithm as an ordered list of labeled steps
Overlay-state indicator — carrier overlay steps visually distinguished from the bureau base
State and edition selector — view the algorithm as it resolves for any specific state and edition
Light edit mode — enable, disable or insert overlay steps without touching the bureau base
Locked-step protection — bureau base steps stay immutable; overlays are saved per carrier
WHAT THIS MEANS IN PRACTICE
→
Actuaries read and verify the algorithm without reading code — bureau correctness becomes auditable, not a black box
→
Customer-specific overlays are visible insertions — not hidden code branches that only engineers can find
→
Insertion errors are caught before any policy is rated — ordering rules enforced by the system
→
Customers test a new overlay against their historical book — before turning it on against new business
→
The same overlay system extends to every line — the customer writes, the platform supports

Bureau-grounded issuance

Quote-to-bind is already core Federato territory. Compliant implementation means the workflow the carrier already runs is now grounded in bureau-correct content end to end, with deviations explicit and circulars traceable.

Regulator-ready bind record — the bound policy carries its full form schedule, step-level trace and the exact content version it rated against

ISO Permissions

LICENSING
BUILT FOR
●FDEs configuring the carrier's ISO Content Licensing Agreement with Verisk
●Engineers onboarding a carrier and scoping its bureau content rights
●Forms and actuarial teams when a cell goes dark and content disappears
THE PROBLEM

ISO content is licensed, not free. Every carrier signs an ISO Content Licensing Agreement with Verisk that scopes redistribution rights to the LOB, the service and the state. Today that licensed scope lives in legal spreadsheets — the rating engine has no way to refuse content the carrier is not entitled to consume, and no way to stop surfacing bureau rates and forms when the agreement lapses. That is a contract risk the platform owes the customer to remove.

WHAT THE PROTOTYPE DOES
ISO Content Licensing Agreement record — in-place flag, as-of and expiration dates, uploaded agreement file, computed status badge per carrier
Per-carrier 2,226-cell matrix — 14 ISO LOBs × 3 services (forms, loss costs, rules) × 53 jurisdictions, every cell independently toggleable
Sliced editor — pick an LOB and a service, flip rights across all jurisdictions at once, instead of facing a wall of 2,226 cells
Two-gate enforcement — agreement gate AND matrix gate, both must pass for ISO content to surface anywhere in the app
Draft, dirty and save — stage a full set of changes before committing, with an explicit dirty indicator for FDE review
WHAT THIS MEANS IN PRACTICE
→
The engine refuses content the carrier is not licensed to consume — contract enforcement becomes systemic, not aspirational
→
An expired ISO Content Licensing Agreement stops bureau content at the gate — no quietly issuing forms past the expiration date
→
FDEs see, in one place, exactly what every carrier is licensed for — no more chasing legal spreadsheets
→
Adding a new licensed LOB is a UI action with an audit trail — not a code change buried in Python
→
The same gate extends to every line — the customer signs an amendment, the FDE flips the cells, the engine respects it

APPENDIX · CARRIER HIERARCHY

One bureau corpus, many writing companies and products

An insurance group writes through multiple writing companies (paper companies), each selling multiple product tiers. The carrier consumes a singular ERC corpus through a tree — and every level above state multiplies the filing, audit and licensing surface.
ORG MODEL
WRITING CO. × PRODUCT × STATE
EXAMPLE
ACME Insurance Group
three writing companies · multiple ISO LOBs · multiple products · 50 states
ACME Insurance Groupparent / legal entity
├─ ACME PreferredNAIC 12345
│ ├─ ISO GL→ GL — PreferredPREFERRED
│ ├─ ISO COMMERCIAL PROPERTY→ CP — Preferred
│ └─ ISO COMMERCIAL AUTO→ Auto — Preferred
├─ ACME StandardNAIC 12346
│ ├─ ISO GL→ Standard + Non-Standard
│ └─ ISO BOP→ BOP — Standard
└─ ACME SpecialtyNAIC 12347
└─ ISO GL→ GL — Specialty (Non-Std)
each (writing co. × LOB × product) → State × Edition (rate / forms / rules)
FILING TRACK = (WRITING CO. × LOB × PRODUCT × STATE)
HOW THIS AFFECTS ISO ERC IMPLEMENTATION
Seven consequences of the tree
01
ERC content stays singular. ISO ships one corpus per (LOB, jurisdiction, edition). There is no Preferred / Standard / Non-Standard axis in the bureau delivery.
02
Tier differentiation is 100% carrier-side. LCM, appetite, form package, schedule-rating cap, UW authority — all of it lives on the carrier-deviation layer. None of it touches the ERC manifest.
03
Filing scope multiplies. Each (writing co. × product × state) is independently filed with its own SERFF tracking number, approval status and potentially its own ERC edition. Tracks drift independently over time.
04
ISO CLM licensing is per-NAIC. Each writing company is a separate ISO licensee. One group-level license is not enough — ISO audits redistribution against the NAIC that issued the form.
05
Audit trail anchors at the NAIC. A DOI examiner asking "which filing bound this policy?" expects writing co. NAIC, product, SERFF tracking number and the three editions in force — not the parent group.
06
Edition resolution is per-filing-track. Each (writing co. × product × state) sits on its own rate / forms / rules editions. The split filing-track resolver extends one level deeper than today.
07
State authorization is per-writing-co. ACME Preferred may not be licensed in NY; ACME Standard may be. Quote / bind gates on the writing co.'s authorization, not the parent's.
ONE BUREAU CORPUS
1
ISO publishes one ERC delivery per (LOB, jurisdiction, edition). Identical content for every NAIC underneath.
ACME FILING TRACKS
~200
4 product tracks × 50 states. Each is independently filed with its own SERFF tracking, approval status and edition cadence.
WHAT MULTIPLIES
× SERFF filings per state
× CLM licenses per writing co.
× State authorizations per NAIC
× Audit-package anchors at bind
× Edition tracks (rate / forms / rules)
…ABOVE THE STATE NODE
THE ONE-LINE
The bureau ships one ERC corpus. The carrier consumes it through a tree — parent → writing co. (NAIC) → product → state → edition — and every node above state multiplies the filing, audit and licensing surface while the bureau content underneath stays singular.
